elasticsearch 学习笔记之 shrink index

Posted on 2020-07-24 | In elasticsearch | Views:

简介

shrink index API 的用途是缩减索引主分片的数量， 它将生成更大的分片，适合于长期存储冷数据。以下内容翻译自官方文档 elasticsearch 6.8 reference - shrink index，并添加了基于 java api 的实现代码，以供日后翻阅。

收缩索引

使用 shrink index API 可以将一个已存在的索引收缩到一个主分片数更少的目标索引中。目标索引的主分片数必须是原索引主分片数的一个因子。例如，如果原索引的主分片数为 8，那么目标索引的允许的主分片数只能为 4、2、1，如果原索引的主分片为一个质数，那么目标索引的主分片数只能为 1。除了主分片数量上的要求外，在收缩请求之前，在某个节点上，原索引的每个分片都要有一个拷贝（可以是副本分片也可以是主分片）分配在这个节点上。

shrink 的内部实现流程如下：

  • 首先，使用与原索引相同的配置创建一个新的目标索引，但其主分片数小于原索引的主分片数。
  • 将源索引的段（segments），硬链接（hardlinks）到目标索引，如果文件系统不支持硬链接，那么只能将段复制到目标索引中，这当然是一个耗时操作。此外，如果原索引配置使用了多数据路径（data path）并且各个路径不在同一个硬盘上，则同样需要拷贝所有的段文件，因为硬链接不支持跨硬盘。
  • 最后，目标索引恢复使用，就像其被重新打开一样。

准备收缩一个索引

为了收缩索引，首先，需要将索引设置为只读。其次，索引的每个分片都要有一个拷贝重新定位到某个节点上并且其健康值为绿色。

这两个前提条件可以使用以下请求实现：

(1) 强制重定位每个分配的一个拷贝到节点 shrink_node_name 上，更多重定位的信息请参考Shard Allocation Filtering。

(2) 阻止索引的写操作，同时仍允许其更改元数据，例如删除索引。

重定位源索引可能需要一段时间，可以使用 _cat recovery API 跟踪进度，或者使用 cluster health API 查看relocating_shards 参数，以等待所有的分配重定位完成。

目标索引成功加入集群后，以上请求会立即返回，而不会等待收缩请求真正完成。

索引只有满足以下所有必要条件才会真正收缩:

  • 目标索引必须不存在
  • 原索引比目标索引拥有更多的主分片数量
  • 目标索引的主分片数量必须是原索引主分片数量的一个因子
  • 原索引的所有分片总文档数必须小于2,147,483,519，因为如果这些分片收缩到目标索引的单个分片上，这个数字是可以放入单个分片的最大文档数。
  • 处理收缩过程的节点必须具有足够的磁盘空间，以容纳现有索引的第二个拷贝。

_shrink API 有点类似于创建索引的 API，目标索引可以设置settings 及 alias 参数：

(1) 目标索引的主分片数，其值必须是原索引分片数的因子。

(2) 最佳压缩仅对索引新写入起效，例如强制合并分片到单个段时。

其它注意事项：

  • _shrink 请求时不可指定 mapping
  • 默认情况下，在_shrink 请求时，除了index.analysis 、index.similarity 以及 index.sort 之外，其余原索引的配置将不会拷贝到目标索引。我们可以通过将 url 参数copy_settings 设置为 true， 使得所有不拷贝的配置拷贝到目标索引中。请注意，copy_settings 不可设置为 false，此参数将在 8.0.0 中移除。

监控收缩过程

收缩过程可以使用 _cat recovery API 监控，或者使用 cluster health API 设置 wait_for_status 为 yellow 来等待完成所有主分片的分配，例如：

_shrink API 在目标索引加入集群状态后立即返回，它不会等待分片分片完成。此时，所有分片都处于 unasigned 状态，基于任何原因无法在收缩节点上分配目标索引，其主分片都将保持未分配状态，直到可以在该节点上分配为止。

一旦住分片分配完成，其状态将转换为 initializing，此时收缩过程将真正开始。当收缩操作结束，分片状态为active。在这之后，Elasticsearch 将尝试分配副本，并可能决定将主分片重定位到另一个节点上。

等待分片可用（active）

由于收缩操作会创建一个新的索引来缩小分片，因此在创建索引时等待活动分片的设置，也适用于收缩索引操作。

elasticsearch 学习笔记之 match_phrase_prefix

Posted on 2019-11-22 | In elasticsearch | Views:

简介

match_phrase_prefix 查询与上一篇笔记记录的 match_phrase 查询相似，返回包含且相对位置与我们提供的文本单词词组一致的文档，不同之处在于match_phrase_prefix 查询文本中的最后一个单词被视为前缀，以该单词开头的任何单词都将匹配。

举个简单的例子：如果我们的搜索文本为 quick brown f ，它将匹配quick brown fox 或 是two quick brown ferrets ，但无法匹配包含 fox is quick and brown。

语法

match_phrase_prefix 的顶层参数：

  • <field> (Required, object) - 表示要搜索的目标文档字段名称

<field> 的参数：

  • query(Required, string) - 表示希望在 <field> 字段里搜索到的文本内容
  • analyzer(Optional, string) - 表示指定一个分析器，用于分析 query 参数输入的搜索文本。如果未指定，则使用索引 field 字段时使用的分析器，如果 field 字段也未指定，则使用索引创建时的分析器。
  • max_expansions(Optional, string) - 用于控制前缀扩展成分词的数量，默认值是 50，这个参数常用于控制返回的匹配文档数量，需要特别注意的是：max_expansions 是作用于分片级别(shard level)的，这意味着即使设置为1，依然有可能匹配到多个词，这些词来自不同的分片（shards）。这种行为使得结果看起来跟 max_expansions 没生效一样，因此谨记计算返回搜索结果的关键词数量不能作为检验 max_expansions是否生效的方法。更多的介绍，请参考博客 How to Use Fuzzy Searches in Elasticsearch 。
  • slop - 用于指定短语之间最大的间隔距离，默认是 0，即必须紧挨着。

elasticsearch 学习笔记之 match_phrase

Posted on 2019-11-21 | Edited on 2019-11-22 | In elasticsearch | Views:

简介

match_phrase 查询将使用特定分析器（通过参数 analyzer 配置）分析输入的搜索文本，并从分析的文本中创建词组查询。换而言之，就是找出匹配一组词条的所有文档， 并且这组词条在文档中出现的顺序必须与其本身的先后顺序一致，各个词条之间的最大相隔距离可以通过参数 slop 配置，默认是紧密相连，即 slop 为 0。

语法

match_phrase 的最上层参数：

  • <field>(Required, object) - 表示要搜索的目标字段

<field> 的参数：

  • query(Required, string) - 表示用户输入的搜索文本
  • analyzer(Optional, string) - 分析器用于分析 query 参数输入的搜索文本。如果未指定，则使用索引 field 字段时使用的分析器，如果 field 字段也未指定，则使用索引创建时的分析器。
  • slop(Optional, integer) - 用于指定短语之间最大的间隔距离，默认是 0，即必须紧挨着。

Java 8 之 lambda 表达式及函数接口

Posted on 2019-07-11 | In Java 基础 | Views:

Lambda 表达式

Lambda 表达式理解为简洁地表示可传递匿名函数的一种方式：它没有名称，但有参数列表、函数主体、返回类型以及可能抛出的异常列表。逐个拆解定义中涉及的概念：

  • 匿名：它无须像普通方法那样有一个明确的名称；
  • 函数： lambda 表达式不像方法那样必须属于某个特定的类，但其形式又与方法一样，有参数列表、函数主体、返回类型；
  • 传递：lambda 表达式可以作为参数传递给方法或存储在变量中；
  • 简洁：无须像匿名类那样写很多模版代码。

简而言之，lambda 表达式是一个可传递的代码块，可以在声明之后使用一次或多次，其形式可以简单的描述为参数列表、箭头、lambda 主体，下面是常见的 lambda 表达式的例子：

使用案例 Lambda 示例
布尔表达式 (List<String> list) -> list.isEmpty()
创建对象 () -> new Apple(10)
消费一个对象 (Apple a) -> {System.out.println(a.getWeigth());}
从一个对象选择或抽取 (String s) -> s.length()
组合两个值 (Int a, int b) -> a * b
比较两个对象 (Apple a1, Apple a2) -> a1.getWeight().compareTo(a2.getWeight())

1 - 概述

TestNG 是一个被设计用来简化广泛的测试需求的测试框架，它既可应用于单元测试（测试一个独立的类）可以用于集成测试（测试一个完整的系统，这个系统可能由多个类、多个包甚至多个外部框架组成，比如多个应用服务）。

使用 TestNG 完成一次测试通常需要 3 个步骤：

  • 在你的代码里书写测试业务逻辑并加入 TestNG 注解.
  • 将你的测试信息（例如：你希望运行测试类名、测试组名等）加入一个 testng.xml 文件或 build.xml 文件.
  • 运行 TestNG

你可以在主页找到快速入门的例子。
在本文档使用到的概念定义如下：

  • 一个 suite 通过一个 XML 文件表示，在文件中使用<suite>标签定义，它可以包含一个或多个测试；
  • 一个测试由 <test> 标签表示，它可以包含一个或多个 TestNG 类；
  • 一个 TestNG 类是至少包含一个 TestNG 注解的 Java 类，它由<class>标签表示，可以包含一个或多个测试方法；
  • 一个测试方法是一个在代码中被 @Test注解的 Java 方法。

一个 TestNG 测试可以使用 @BeforeXXX和 @AfterXXX 注解配置，这些注解允许在某个点之前或之后执行一些 Java 逻辑，这些点可以是上述列表中任一项。

概述

收集器，一种通用的、从流生成复杂值的结构。只要将它传给 collect方法，所有流都可以使用它。本文将介绍标准类库里提供的常用收集器及其它们的组合使用方式，以此简化某些代码的编写方式。此外，收集器框架也提供了非常通用的API，供我们定制自己的收集器，不过在决定实现收集器之前，应仔细阅读 API 文档是否已存在满足条件的收集器，避免重复造轮子。

标准类库里的收集器

对Collectors里常用的收集器进行简单的分类，以助记忆使用：

  • 转换成其它集合：toList、toSet、toCollection、toMap、toConcurrentMap
  • 生成一个值：maxBy、minBy、counting、summingInt、averagingInt、summarizingInt，针对后面 3 种，还有 long、double 的重载版；

  • 针对 String 类的特殊收集器：joining，这个收集器用于拼接字符串，并可自定义前缀、分隔符、后缀；
  • reducing：上述各种特殊收集器的一般化，即理论上可以使用 reducing 替换上面这些特殊的收集器；
  • 组合收集器：多个收集器可以进行组合使用

Java 8 之接口默认方法及静态方法

Posted on 2019-02-18 | In Java 基础 | Views:

概要

Java 8 新增了很多语言特性，其中涉及接口的有默认方法、静态方法，这些语言特性旨在帮助编写类库的开发人员，但对我们日常写应用程序的开发人员也同样适用。

默认方法

默认方法，简而言之，就是在一个接口里使用 default 关键字定义的一个方法，所有该接口的实现类都可以直接使用而无须自己实现此方法。

默认方法存在的主要目的是为了方便扩展已有接口，如果没有默认方法，给一个已经发布的接口添加方法时，所有实现改接口的地方都得做修改，如果该接口的使用范围很广，那相应的影响也非常之大，例如 JDK 里定义的各种接口。因此，Java 8 中加入了此语言特性使得类库设计人员可以很方便的扩展已有的接口。

重写规则

实现包含默认方法的接口，有以下三种可能的方式：

  • 实现类不重写默认方法，直接使用接口中的默认方法；
  • 子接口或抽象类将默认方法声明为普通的抽象方法；
  • 实现类中重写默认方法，与重写普通的接口方法一样。

Elasticsearch Client 使用介绍

Posted on 2018-09-30 | In elasticsearch | Views:

基本概念

Elasticsearch Client 指的是 Elasticsearch 的 Java API 所提供的客户端，我们以多种方式使用 Client：

  • 在已有的集群上执行标准的 index、get、delete、search 操作
  • 在正在运行的集群上执行管理任务

获取 Elasticsearch Client 很简单，最常用的方式是创建TransportClient实例。TransportClient 使用 transport 模块远程连接到 Elasticsearch 集群，它自身不加入集群，而是简单地获取一个或多个初始传输地址，并在每个操作上以轮询的方式与集群通信。

需要注意的是官方文档中说明 6.x 及其以下版本可以正常使用 TransportClient，在 7.x 会将其废弃，而到 8.x 则会彻底废除，因此在使用需要注意 Elasticsearch 客户端 API 的版本以及后续的迁移计划。
