1 - 概述

TestNG 是一个被设计用来简化广泛的测试需求的测试框架，它既可应用于单元测试（测试一个独立的类）可以用于集成测试（测试一个完整的系统，这个系统可能由多个类、多个包甚至多个外部框架组成，比如多个应用服务）。

使用 TestNG 完成一次测试通常需要 3 个步骤：

  • 在你的代码里书写测试业务逻辑并加入 TestNG 注解.
  • 将你的测试信息（例如：你希望运行测试类名、测试组名等）加入一个 testng.xml 文件或 build.xml 文件.
  • 运行 TestNG

你可以在主页找到快速入门的例子。
在本文档使用到的概念定义如下：

  • 一个 suite 通过一个 XML 文件表示，在文件中使用<suite>标签定义，它可以包含一个或多个测试；
  • 一个测试由 <test> 标签表示，它可以包含一个或多个 TestNG 类；
  • 一个 TestNG 类是至少包含一个 TestNG 注解的 Java 类，它由<class>标签表示，可以包含一个或多个测试方法；
  • 一个测试方法是一个在代码中被 @Test注解的 Java 方法。

一个 TestNG 测试可以使用 @BeforeXXX和 @AfterXXX 注解配置，这些注解允许在某个点之前或之后执行一些 Java 逻辑，这些点可以是上述列表中任一项。

本手册剩余的内容将介绍以下内容：

  • 一个介绍所有注解简要含义的列表，它可以给你关于这些注解丰富功能的初步印象，如果需要探究这些注解的细节知识，你可能需要进一步查阅专门针对相关注解的文章；
  • 关于 testng.xml 文件的描述，它的语法以及可以在其中指定的内容；
  • 一份详细的特性列表以及如何结合注解、testng.xml中使用它们的说明。

2 - 注解

我们快速预览以下 TestNG 中的注解及其它们的属性。

2.1 @BeforeXXX 及 @AfterXXX 注解

用于一个TestNG类的信息配置的注解
@BeforeSuite 被注解的方法将在其所在的 suite 中的所有 test 运行之前执行，注：被注解的方法可以属于此 suite 下的任何一个类中
@AfterSuite 被注解的方法将在其所在的 suite 中的所有 test 运行之后执行
@BeforeTest 被注解的方法将在<test>标签下所有类的所有方法执行之前执行
@AfterTest 被注解的方法将在<test>标签下所有类的所有方法执行之后执行
@BeforeGroups 被注解的方法将在该组别下所有方法执行之前执行
@AfterGroups 被注解的方法将在该组别下所有方法执行之后执行
@BeforeClass 被注解的方法将在当前类第一个方法执行之前执行，注：如果在@Test使用 groups属性时，@BeforeClass 会无法执行，参考解决方案
@AfterClass 被注解的方法将在当前类第一个方法执行之后执行
@BeforeMethod 被注解的方法将在其所属类的每一个测试方法执行之前执行
@AfterMethod 被注解的方法将在其所属类的每一个测试方法执行之后执行

上述注解在 TestNG 父类中的表示
当在一个 TestNG 类的父类中加入上述注解时，相应的注解行为将在 TestNG 类中被继承。当需要对多个测试类的通用父类进行集中的测试设置时，继承功能将很有用。在继承的情况下，TestNG 将保证@Before 的方法按继承顺序执行（顶层父类优先，然后沿着继承链向下执行），@After 的方法逆序执行（按继承链反向执行）。

上述注解共通参数说明

名称 含义
alwaysRun 对于@BeforeXXX（除了@BeforeGroups）： 不论被配置方法属于哪个群组，它都会执行；对于被@AfterXXX注解的方法，当该参数设置为 true 时，即使前面触发的方法执行失败或者被跳过，被配置方法都会执行
dependsOnGroups 需要依赖的群组列表
dependsOnMethods 需要依赖的方法列表，此参数用于定义执行顺序，dependsOnMethods 是强依赖，dependsOnGroups 是弱依赖
enabled 类或方法是否激活，值为 false 时，该方法不执行
groups 类或方法所属的群组列表
inheritGroups 如果值为 true，这个方法将属于类级别中被 @Test注解所指定的群组

2.2 @DataProvider 注解

此注解将一个方法标记作为测试方法（使用@Test注解的方法）的数据源，被标记的方法需返回一个Object[][],其中每一个Object[]都是测试方法的一份参数列表。想要接收数据源的测试方法需要使用dataProvider参数，其值与@DataProvider的参数name的值相同。

注解参数：

  • name：数据源的唯一名字，供测试方法通过名字引用数据源，如果该参数没有被提供，将被自动设置为这个被此注解标识的方法的名称。
  • parallel:如果设置为 true，则与测试方法并行执行，默认值为 false。

2.3 @Factory 注解

此注解将一个方法标记为工厂方法，其返回的值将在 TestNG 的测试类中使用，返回值类型必须为 Object[]。

2.4 @Listeners 注解

在测试类中定义监听器，参数value的值为一组继承了 org.testng.ITestNGListener的类型。

2.5 @Parameters 注解

描述如何将参数传递给测试方法，参数value的值为填充该方法的参数的变量列表。

2.6 @Test 注解

将一个方法/类标记为测试的一部分。
注解参数说明：

名称 含义
alwaysRun 如果设置为true，即使这个测试方法所依赖的方法运行失败，它仍然会运行
dataProvider 用于标注这个测试方法的 data provider 的名称
dataProviderClass 标注其data provider所在的类。如果没有使用该参数，测试方法将在其所在类或其父类中寻找data provider；如果这个参数被使用了，测试方法的data provider方法需要是该类中的静态方法
dependsOngroups 测试方法所依赖的组列表
dependsOnMethods 测试方法所依赖的方法列表
description 测试方法的描述信息
enabled 关于这个类/方法的其他方法是否可用
expectedExceptions 期望测试方法抛出的异常列表。如果没有异常或者一个不同于该列表的异常被抛出，该测试将被标记为失败
groups 这个类/方法所属的组列表
invocationCount 这个测试方法应该被触发的次数
invocationTimeOut 这个测试方法被全部触发的所有累计时间的最大值，以毫秒记；如果 invocationCount 未使用，此属性将被忽略
priority 这个测试方法的优先级，低级别的方法将优先被调用
successPercentage 这个测试方法期望运行成功的百分比
singleThreaded 如果设置为true，这个测试类里的所有方法将在同一个线程中执行，即使这些方法带着参数parallel=”method”在运行；这个属性只能被使用在类级别，如果被使用在方法级别将会被忽略
timeOut 测试消耗时间的最大值，以秒记
threadPoolSize 测试方法的线程池数量，这个测试方法将被以invocationCount标记的多个线程触发；PS：如果invocationCount未设置，此属性将被忽略

5.2 测试组别

TestNG 允许你执行一些复杂的测试方法组，这意味着你不仅可以声明方法属于哪个组，还可以自定义组别，并在其中包含已有的组别。然后，TestNG 可以被要求执行某些包含的指定组别而不执行另一些不包含的组别。这个特点让我们能方便的分割你的测试方法，并且可以在不再次编译的前提下连续执行两个不同的测试组。

组别是在 testng.xml 中定义的，它可以定义在<suite>或<test>标签下。组别定义在<suite>标签下可以应用到其下的<test>标签。需要注意的是，组别在这些标签下累计的：假如你在<suite>标签下指定了组别”a”，在<test>标签指定了组别”b”，那么组别”a”跟组别”b”都会被包含。

5.10.2 并行测试集、类、方法

在<suite>标签上的 parallel 属性可以有以下值：

  • parallel=”methods”:TestNG 将在不同的线程中运行每个测试方法，依赖的方法也将在不同的线程中运行，但它们将遵循你指定的顺序。
  • parallel=”tests”：TestNG 会使用相同的线程运行同一个<test>标签下的所有测试方法，但是每一个<test>标签都会在一个独立的线程中运行。这使得你可以将所有不是线程安全的类组合起来放在同一个<test>中，并确保它们都将在同一个线程中运行，同时又不会失去 TestNG 用多个线程运行测试方法所带来的好处。
  • parallel=”classes”：TestNG 将在同一个线程中运行同一个类中的所有方法，但是每个类将在一个单独的线程中运行。
  • parallel=”instances”:TestNG 将在同一个线程中运行同一个实例中的所有方法，但是两个不同实例上的两个方法将运行在不同的线程中。

此外，thread-count 属性允许你指定应为此次执行分配多少个线程。

  • 任何 @Before方法或 @Test 方法可以声明 ITestContext类型的参数；

  • 任何 @DataProvider方法可以声明 ITestContext 或 java.lang.reflect.Method 类型的参数。后者将接收即将执行的那个测试方法。

一个典型的例子是，尝试利用您现有的套件文件，并尝试使用它来模拟您的“应用程序测试”中的负载测试。在最小程度上，您将会多次复制您的标签的内容，并创建一个新的suite xml文件并使用它。但这似乎并没有太大的规模。

TestNG允许您在运行时通过监听器更改在您的套件xml文件中的一个套件(或)一个测试标记。您可以通过提供一个实现 IAlterSuiteListener的监听器来实现这一点。请参考 TestNG 监听器了解相关的监听器。

6.2 记录与结果

测试运行的结果是在一个名为index.html的文件中创建的，这个文件所在的目录是在启动SuiteRunner时指定的。这个文件指向各种各样的 HTML 及文本文件，这些文件包含了整个测试运行的结果。

使用 TestNG 的Listeners与Reporters很容易生成你自己的测试报告：

  • Listeners 实现了org.testng.ITestListener接口，在运行时当一个测试启动、通过、失败时会得到通知。
  • Reporters 实现了 org.testng.IReporter接口，当所有的套件都由 TestNG 运行时，会得到通知。IReporter实例会接收到一个描述整个测试运行的对象列表。

假如你想生成 pdf 格式的测试报告，你不需要实时的得到通知，因此你应该使用IReporter。如果您想要编写测试的实时报告，例如一个带有进度条的 GUI，或者一个文本报告显示点(“.”)，当每个测试被调用时(如下所示)，ITestListener是您最好的选择。
